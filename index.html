Architecture

系统架构总览

整体架构分为四层：Agent 编排层 → Context 组合层 → LLM 推理层 → 基础数据层。每层职责清晰，小模型只做"阅读理解"式的匹配工作，复杂推理由上层预先完成。

Agent Layer  ·  智能编排层
问诊管理 · RAG 编排 · 推理链 · 质量校验
Inquiry Mgr → RAG Orchestrator → Reason Chain → QA
Context Composition Layer  ·  上下文编排层
将检索结果编排为 LLM 可消费的 Structured Context
LLM Inference (Edge)  ·  本地推理层
Local Small Model (~7B) · 不微调 · 不记忆
Knowledge Base  ·  知识库层
症状标准化引擎 · 证型 KB · 鉴别诊断矩阵 · 方药库

核心设计理念

质量排序：数据质量 > Context 编排质量 > Agent 推理链设计 > 模型大小

Context 中 避免让 LLM 做回忆/检索类工作——所有事实性信息已在 context 中。只让 LLM 做"匹配"和"组织"——这是小模型相对擅长的。

Data Layer

RAG 数据层设计

数据质量是整个系统的基础。我们分为三个核心子模块构建。

① 症状标准化层

病人主诉是自然语言，必须先映射到标准 TCM 术语。此环节完全不经过 LLM，由规则引擎 + embedding 检索完成。

// 输入：病人自然语言主诉 "这几天头晕，晚上睡不好，腰酸" // 症状标准化模块（规则匹配 + embedding 检索） // ↓ // 输出：结构化标准症状 { "primary_symptoms": ["头晕"], "secondary_symptoms": ["失眠", "腰膝酸软"], "tongue": "待采集", "pulse": "待采集", "status": "incomplete" }

② 证型知识库结构

这是 RAG 的核心数据集。每条记录包含以下字段：

{ "pattern_id": "shen_yin_xu", "pattern_name": "肾阴虚证", "category": "脏腑辨证", "core_symptoms": ["头晕", "耳鸣", "腰膝酸软"], "secondary_symptoms": ["失眠", "盗汗", "五心烦热", "口干咽燥"], "tongue": "舌红少苔", "pulse": "脉细数", "etiology": "久病伤肾 / 房劳过度 / 过服温燥", "treatment_principle": "滋补肾阴", "representative_formula": "六味地黄丸加减", "key_differentiation": { "vs_shen_yang_xu": "肾阴虚有热象（五心烦热、盗汗），肾阳虚有寒象（畏寒肢冷）", "vs_fei_yin_xu": "肾阴虚以腰膝症状为主，肺阴虚以咳嗽咽干为主" } }

③ 鉴别诊断矩阵（关键差异化设计）

这是让结果质量优于裸小模型的关键。为每个证型预计算鉴别路径：

证型 核心鉴别点 相似证型 区分关键
肾阴虚 五心烦热、盗汗、舌红少苔 肾阳虚 寒热之别：畏寒 vs 烦热
肾阳虚 畏寒肢冷、小便清长、舌淡胖 肾气虚 寒象有无：畏寒 vs 气短乏力
肝郁气滞 胁肋胀痛、情志抑郁、脉弦 肝火炽盛 火象有无：胀痛 vs 灼痛、面红目赤
心脾两虚 心悸失眠、食少便溏、面色萎黄 心肾不交 腰膝症状＋热象有无
痰湿阻肺 咳嗽痰多、胸闷、苔白腻 风寒束肺 表证有无：恶寒发热 vs 无表证

检索时同时返回相似证型及其区分要点，小模型不需要"自己推理出差异"，而是直接拿到对比表。

数据层构建建议

01

症状→证型映射矩阵

每个标准症状指向哪些潜在证型及其权重。这是 embedding 检索的基础。

02

证型鉴别诊断表

证型之间的相似度矩阵与鉴别要点。确保鉴别诊断的完整性。

03

证型→方药对照

标准方剂、加减法、禁忌配伍。保证治疗建议的可靠性。

Reasoning Chain

Agent 推理链设计

小模型不能做复杂推理，所以我们将推理摊平为多个简单步骤，每步只做一种判断。

症状收集与整理

输入：病人主诉 + 追问回答  →  输出：结构化症状清单 {主症, 次症, 舌, 脉, 其他}

不依赖 LLM 完全由规则 + 追问模板完成。

候选证型粗筛 (Top-K Retrieval)

Query： [主症 embedding] + [次症 embedding]  →  检索： Top-5 候选证型 + 匹配度分数

不依赖 LLM 向量数据库语义检索，只看症状层面的相似度。

细筛与鉴别（LLM 首次参与）

检索候选证型的鉴别诊断信息，构造对比表：

症状肾阴虚肾阳虚肾精不足
畏寒✗✓✗
五心烦热✓✗✗
腰膝酸软✓✓✓
发育迟缓✗✗✓

LLM 参与 让模型基于对比表选择最匹配的证型。任务从"复杂辨证推理"降级为"表格阅读理解"。

治疗建议生成

检索选中证型的治则治法 + 标准方药，LLM 基于患者具体情况做个性化微调（加减化裁）。

LLM 参与

为什么这么做有效

任务的认知复杂度从 Bloom 分类学的高阶（辨证推理、综合分析）降阶为 低阶（信息提取、模式匹配）。小模型（~7B）在模式匹配任务上的表现通常接近大模型。

Prompt Engineering

Context 编排设计

这是 RAG 与 LLM 之间的桥梁。编排后的 Context 模板如下：

【系统指令】 你是一个中医诊断助手。请基于以下患者信息和诊断数据， 进行辨证分析。 【患者信息】 主症：{头晕, 耳鸣} 次症：{失眠, 腰膝酸软, 口干咽燥} 舌象：舌红少苔 脉象：脉细数 【候选证型分析】 | 证型 | 匹配度 | 核心符合 | 不符合 | |----------|--------|----------|------------| | 肾阴虚 | 92% | 头晕, 腰膝酸软, 舌红少苔, 脉细数 | - | | 肾阳虚 | 35% | 腰膝酸软 | 无寒象 | | 肾精不足 | 50% | 耳鸣, 腰膝酸软 | 无发育/生殖问题 | 【鉴别诊断要点】 - 肾阴虚 vs 肾阳虚: 肾阴虚有热象(五心烦热、盗汗), 肾阳虚有寒象(畏寒肢冷) - 肾阴虚 vs 肾精不足: 肾阴虚有虚热, 肾精不足无热象, 以发育/生殖功能减退为主 【参考标准】肾阴虚证诊断标准: - 主症 ≥2 项: {头晕, 耳鸣, 腰膝酸软, 遗精} - 次症 ≥2 项: {失眠, 口干, 盗汗, 五心烦热} - 舌脉: 舌红少苔, 脉细数 【任务】 1. 分析患者症状符合哪个证型 2. 给出辨证依据 3. 提出治则治法 4. 建议基础方药 请严格按以下 JSON 格式输出： {{ "diagnosis": "...", "evidence": "...", "principle": "...", "formula": "...", "modifications": "...", "confidence": 0-1 }}
①

事实先行

所有需要模型"回忆"的知识（证型标准、症状关联、方剂组成）全部放在 context 中。模型只需要做"匹配"。

②

结构化数据

对比表、匹配度分数等结构化信息比段落文本更容易被小模型理解。表格格式优于叙述。

③

输出格式约束

JSON 格式 + 固定字段 → 方便后续规则校验 → 不依赖模型自由生成能力。

④

不诱导幻觉

避免"请详细分析"类开放指令。使用"请选择"而不是"请生成"。关闭模型"创造性回答"倾向。

Quality Assurance

质量保障层设计

诊断建议不能出错。我们使用规则-based 的 QA 校验来拦截明显错误——完全不依赖 LLM。

✓

完整性校验

验必填字段：证型、辨证依据、治则、方药是否齐全。缺一不可。

✓

配伍合理性校验

证型与方药是否配伍合理？如肾阴虚不能用附子、肉桂等大热之品。

✓

舌脉一致性校验

舌象、脉象与证型是否一致？如舌淡胖苔白滑 + 肾阴虚 → 矛盾检测。

✓

置信度评估

基于信息完整度给诊断结果打分。缺舌象/脉象时自动降级为"疑似诊断"。

最小完备信息集

必需：主症、舌质、舌苔、脉象

强烈建议：次症（≥2）、二便、饮食、睡眠

可选：月经（女性）、既往史、过敏史

Agent 的第一轮交互应该是追问缺失信息。信息完整度直接决定诊断置信度。

Analysis

可行性分析

优势因素

✓

辨证的结构化特征

证型是有限集合（约 200-400 个基础证型），本质是 pattern matching，非常适合 RAG + 规则引擎。

✓

推理链拆分

复杂辨证拆解为 4 个简单步骤，每步只做小模型能做的事。认知负载大幅降低。

✓

Context 编排补足短板

小模型最弱的知识记忆、复杂推理全部由 context 补足。模型只需做模式匹配。

✓

规则校验保障底线

规则-based QA 层可拦截绝大多数明显错误，保证诊断建议的基本可靠性。

✓

主动追问机制

Agent 可系统性地询问舌象、脉象等缺失信息，提高诊断完整度和置信度。

✓

可增量迭代

数据层独立于模型层，新增证型、优化鉴别逻辑不依赖模型更新。迭代成本低。

关于"Edge Mode + 小模型"的可行性

推荐模型（根据硬件条件选择）：

  • Qwen2.5-7B-Instruct — 中文场景首选，指令跟随能力强
  • Yi-1.5-6B-Chat — 中文医学领域表现较好
  • DeepSeek-V2-Lite-Chat — 推理效率高，适合低配硬件
  • InternLM2-7B — 上海 AI Lab 出品，中文医学术语覆盖好

如果硬件只能跑 3B 以下模型，建议退一步将定位改为"诊断建议助手"而非"诊断结论系统"，同时加强追问层和信息完整度校验，控制输出置信度。

Embedding 模型选型建议：不要用通用 embedding。TCM 术语的语义空间很特殊。优先选择中文医学领域专用 embedding（如 PULSE、华佗系列的 embedding 模型），或使用领域数据自行微调。

Recommendations

关键建议

基于上述设计，以下是实施时最需要关注的优先级：

  1. 数据质量是第一优先级 症状→证型映射矩阵的准确性和覆盖率直接决定系统上限。先花 70% 的精力把数据做扎实，再考虑模型选择和编排优化。
  2. 鉴别诊断矩阵不可省略 这是让小模型"超越自己"的关键。预计算相似证型的区分要点，通过 context 直接给到模型，比让模型自己推理可靠得多。
  3. 信息采集先于诊断 Agent 第一轮必须是追问：舌象如何？脉象如何？二便？月经？信息完整度是置信度的前提。设计"最小完备信息集"的强行入流程。
  4. 规则 QA 层不可或缺 所有校验逻辑用规则实现，不依赖 LLM。检查证型-方药一致性、舌脉-证型一致性、字段完整性。这是防止误诊的最后防线。
  5. 输出定位为"辅助参考" 系统定位应该是"中医师决策辅助工具"。输出中明确标注置信度、缺失信息提示，在 UI 上传达"仅供参考"的信号，不给模型超出实际能力范围的责任。

最终结论

在条件约束下（Edge Mode、小模型、不可微调），可以有好的结果。质量前提是：数据层 > 编排层 > 模型层。

做好这三件事，可以达到接近可用的辅助诊断水平：

① 结构化的证型知识库（含鉴别诊断信息）
② 多步推理链 + 信息采集流程
③ 规则-based QA 校验层

如果把数据层做到 90 分，编排做到 80 分，模型 60 分就够用。
